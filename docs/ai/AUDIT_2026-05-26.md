# AUDIT — Logopädie Report Agent

> **Ist-Zustand-Analyse mit Verbesserungsvorschlägen**
> Doppelperspektive: erfahrene Logopädin (klinische Fachlichkeit) + Senior Software-Entwickler (technische Qualität).
> Erstellt: 2026-05-26 · Auditor: Claude Code · Methode: **Voll-Live-Test** mit echten Groq-Calls + Code-Review + Test-Suiten.

---

## 1. Executive Summary

Der Logopädie Report Agent ist ein **fachlich erstaunlich reifes** Portfolio-Projekt: Die Anamnese-Engine, die Berichtstypen und die Domänenmodellierung (Indikationsschlüssel SP1–SP6, ICD-10, ICF-Codes, §-Bezüge zu HeilM-RL/SGB V) zeigen echtes logopädisches Fachverständnis. Die Code-Basis (~11.900 LOC Backend + Frontend) ist modular, getypt und sauber (ruff/eslint/tsc/mypy laufen fehlerfrei, 311 Backend- + 146 Frontend-Unit-Tests grün).

**Aber das Projekt ist deutlich über seine Dokumentation hinausgewachsen** — es ist inzwischen ein **Multi-User-System mit Auth, 2FA, Patientenverwaltung und Admin-Funktionen**, ohne dass CLAUDE.md, `docs/ai/*` oder die E2E-Tests das nachvollzogen haben. Daraus resultieren die schwerwiegendsten Befunde:

| Schweregrad | Anzahl | Kern-Themen |
|---|---|---|
| 🔴 **Kritisch** | 2 | Broken Access Control auf Sessions (Datenexfiltration); halluzinierte klinische Fakten in Berichten |
| 🟠 **Hoch** | 4 | Rate-Limiting in Produktion wirkungslos; anonyme KI-Nutzung; CI/E2E-Suite rot; Anamnese-Daten ohne Owner-Bindung |
| 🟡 **Mittel** | 6 | Altersgruppen-Bug, klinische Genauigkeitsfehler, 500 statt 422, Doku-Drift, Anamnese-Abschlusslogik, fehlender Disclaimer |
| 🟢 **Niedrig** | 5 | Stale Tests, Modell-Liste, Materials-Truncation, Health-Token-Design, Naming |

**Gesamturteil:** Solides Fundament mit hohem Fachanspruch, aber mit **einem kritischen Sicherheitsleck**, das vor jeder echten Nutzung mit Patientendaten geschlossen werden muss, und einem **systematischen KI-Halluzinationsmuster**, das für medizinische Dokumente ein Haftungs- und Vertrauensrisiko darstellt.

## 4. Befunde nach Schweregrad

### 🔴 KRITISCH

#### C-1 — Broken Object Level Authorization auf Sessions (Datenexfiltration)
**OWASP API #1 (BOLA/IDOR).** Alle Session-Routen in [routers/sessions.py](../../backend/routers/sessions.py) verwenden `get_optional_user` und binden den User nur als `_` (ignoriert) — es gibt **keine** Prüfung `session.user_id == current_user.id`. Die Session wird allein über `store.get(session_id)` geladen ([sessions.py:96-107, 151-180, 275-336](../../backend/routers/sessions.py#L96)).

**Live verifiziert:**
- **A.** Ein **vollständig unauthentifizierter** Request liest die komplette fremde Session inkl. aller klinischen Anamnese-Daten.
- **B.** User B kann in die Session von User A schreiben (`POST /chat`).
- **C.** User B kann den Bericht von User A neu generieren (Patientendaten „Max M.").
- **D.** Der so erzeugte Report landet in **User Bs eigener Report-Liste** (`id:9`, pseudonym „Max M.") → **fremde Patientendaten werden in den eigenen Account exfiltriert.**

Einzige Barriere ist die 48-Bit-Session-ID (12 Hex-Zeichen) als „Capability-URL" — und Sessions sind anonym erzeugbar (siehe H-2).

**Impact:** Vertraulichkeitsbruch klinischer Daten, DSGVO-relevant, sobald reale (auch pseudonymisierte) Patientendaten verarbeitet werden.

**Fix:**
1. Session-Modell um `user_id` erweitern (falls nicht vorhanden bei Erstellung setzen) und in **jeder** Session-Route prüfen: `if session.user_id and session.user_id != current_user.id → 403/404`.
2. Für den bewussten Demo-Modus: anonyme Sessions als `is_demo=true` markieren und **strikt** von authentifizierten Sessions trennen — Demo-Sessions dürfen nie in eine echte User-Report-Liste schreiben.
3. `generate` darf den Report nur dann an `current_user` hängen, wenn dieser auch Owner der Session ist.

---

#### C-2 — KI halluziniert konkrete klinische Fakten in Berichten
In **jedem** generierten Artefakt erfand das Modell präzise, nicht eingegebene klinische Parameter:

| Artefakt | Halluzinierte Angabe (nicht vom Therapeuten geliefert) |
|---|---|
| Befundbericht `empfehlung` | „mindestens einmal wöchentlich, über einen Zeitraum von mindestens 6 Monaten" + „kann sich negativ auf seine Schullaufbahn auswirken" |
| Therapieplan | „Reduzierung … **um 50 %** innerhalb der ersten 6 Sitzungen", „Selbstkorrekturrate **um 20 %**", „12 Sitzungen" |
| Suggest | „mit einer Frequenz von **einmal wöchentlich**" |

**Ursache:** Die Prompt-Struktur in [services/report_generator.py:52](../../backend/services/report_generator.py#L52) fordert aktiv Felder wie *„Empfehlung bezüglich Therapiefrequenz, -dauer"* an, **ohne** zu konditionieren, ob diese Daten erhoben wurden. Die schwache Anweisung *„Bei fehlenden Informationen schreibe 'Nicht erhoben'"* ([report_generator.py:29](../../backend/services/report_generator.py#L29)) verliert gegen die strukturierte Feldabfrage. Bemerkenswert: der **Legacy**-Prompt enthält das starke *„Never invent or guess information"* ([groq_client.py:43](../../backend/services/groq_client.py#L43)) — der Hauptgenerator nicht.

**Impact:** In einem medizinisch-rechtlichen Dokument (Verordnungsbericht geht an Arzt/Kasse) sind erfundene Frequenz-/Dauer-/Prozentangaben ein **Haftungs- und Plausibilitätsrisiko**. Verstößt zudem gegen die globale Projektregel „NIEMALS Faktenwerte erfinden".

**Fix:**
1. Anti-Fabrikations-Guardrail in `_BASE_PROMPT` aufnehmen (analog Legacy): *„Erfinde keine Frequenzen, Dauern, Sitzungszahlen, Prozentwerte oder Testergebnisse. Wenn nicht erhoben: 'Nicht erhoben' oder leer lassen."*
2. Felder wie `empfehlung`/Therapiefrequenz nur befüllen lassen, wenn entsprechende Anamnese-Felder vorhanden sind (Prompt-Konditionierung oder Post-Processing).
3. Optional: nachgelagerter „Fact-Guard"-Check, der Zahlenangaben markiert, die in den Eingabedaten nicht vorkommen.

### 🟠 HOCH

#### H-1 — Rate-Limiting in der Vercel-Produktion praktisch wirkungslos
[middleware/rate_limiter.py:14-26](../../backend/middleware/rate_limiter.py#L14) fällt auf `memory://` zurück, weil Upstash-REST nicht slowapi-kompatibel ist; echtes Redis nur bei gesetztem `RATE_LIMIT_REDIS_URL`. Auf Vercel-Serverless ist In-Memory-Limiting **pro Instanz und flüchtig** (Reset bei jedem Cold Start, nicht instanzübergreifend). CLAUDE.md behauptet „Redis-backed slowapi rate limiting" — faktisch nicht der Fall. `key_func=get_remote_address` ignoriert zudem `X-Forwarded-For` hinter dem Vercel-Proxy.
**Fix:** slowapi-kompatiblen Redis-Backend bereitstellen (z. B. Upstash über `redis://`-Protokoll statt REST) oder einen eigenen Upstash-REST-Limiter implementieren; `X-Forwarded-For`-Auflösung ergänzen. Andernfalls Doku korrigieren und Limit-Versprechen relativieren.

#### H-2 — Anonyme KI-Nutzung ohne Auth (Kosten-/Missbrauchsvektor)
`POST /sessions` und Folgerouten sind via `get_optional_user` **ohne Login** nutzbar (Demo-Modus). Kombiniert mit H-1 (kein wirksames Limit) kann ein anonymer Besucher beliebig echte Whisper-/Llama-Calls auslösen → Groq-Kosten / DoS-Kostenrisiko.
**Fix:** Demo-Modus hart deckeln (server-seitiges Tageskontingent pro IP über persistenten Store), Demo-Sessions kurz-TTL, ggf. CAPTCHA/Proof-of-Work vor dem ersten KI-Call.

#### H-3 — CI- und E2E-Suite ist rot
Live-Lauf: **64 E2E-Tests → 30 grün, 34 rot.** Davon:
- **32 Firefox-Fehler:** `playwright.config.ts` enthält jetzt ein `firefox`-Projekt, aber CI installiert nur `npx playwright install chromium` ([.github/workflows/ci.yml:131](../../.github/workflows/ci.yml#L131)) und `npx playwright test` läuft über **beide** Projekte → Firefox-Binary fehlt, Job schlägt fehl. Lokal identisch.
- **2 Chromium-Fehler:** stale Tests (siehe L-1).

HANDOFF.md/CURRENT.md behaupten „All 32 Chromium e2e tests pass" + „CI is green" — **nicht mehr zutreffend.**
**Fix:** Entweder `npx playwright install firefox chromium --with-deps` in CI ergänzen, oder das Firefox-Projekt entfernen, bis es unterstützt werden soll. Stale Tests fixen.

#### H-4 — Bericht generierbar aus unvollständiger Anamnese
`POST /sessions/{id}/generate` prüft `is_anamnesis_complete` nicht — im Test ließ sich ein vollständiger Bericht erzeugen, während die Engine die Anamnese noch als unvollständig führte. Begünstigt C-2 (Lücken werden halluziniert gefüllt).
**Fix:** Generate gaten oder dem Nutzer fehlende Pflichtfelder explizit anzeigen und im Bericht als „Nicht erhoben" kennzeichnen.

### 🟡 MITTEL

#### M-1 — Altersgruppe wird nicht aus dem Geburtsdatum berechnet
Patient mit `birthdate: 2020-01-15` (≈6 J.) erhält `age_group: "erwachsen"` (Default aus [report_generator.py:165](../../backend/services/report_generator.py#L164) bzw. Patient-Service). Klinisch falsch und steuert Prompt-Logik (Kind- vs. Erwachsenen-Anamnese).
**Fix:** `age_group` aus `birthdate` ableiten (Kind < 18 / Erwachsen), Default nicht „erwachsen".

#### M-2 — Klinische Genauigkeitsfehler der KI-Analyse
- Phonologie: „Hund"→„Tund" als *„Vorverlagerung /h/→/t/"* gelabelt — /h/ ist kein velarer Laut, kein Fronting. Das Modell presst Beobachtungen in bekannte Prozess-Labels.
- `age_appropriate: true` für ein 5;4-jähriges Kind mit Cluster-Reduktion + finaler Konsonantenauslassung — diese Prozesse sollten bis 5 J. weitgehend abgebaut sein; fachlich fragwürdig.
- SOAP `subjective`: „Max M. berichtet …" bei einem 5-Jährigen — Subjective sollte Elternangabe sein.
**Fix:** Prompt-Constraints (nur belegbare Prozesse labeln, Alters-Normtabellen referenzieren, Subjective alters-/quellenkorrekt). Klinische Review-Schleife für die Analyse-Prompts.

#### M-3 — `500` statt `422` bei phonologischer Textanalyse
[services/phonological_analyzer.py:93](../../backend/services/phonological_analyzer.py#L93) greift auf `p["production"]` zu; das Request-Schema ist nur ein untypisiertes `list[dict[str,str]]`. Falscher Key → unbehandelter `KeyError` → **HTTP 500** statt sauberem `422`.
**Fix:** Pydantic-Modell `WordPair{target: str, production: str}` definieren; FastAPI liefert dann automatisch 422 mit klarer Meldung.

#### M-4 — Dokumentations-Drift (Quelle-der-Wahrheit verletzt)
CLAUDE.md beschreibt „middleware/auth.py = Optional API-Key auth", „Groq", „35 Tests", „9 Router" — real: JWT-Auth + ServiceToken-Middleware, Auth/Patienten/Admin/2FA, 311 Tests, 13 Router. `docs/ai/CURRENT.md`/`HANDOFF.md` behaupten grüne CI. AGENTS.md erhebt die Repo-Dateien zur „shared truth" — der Drift untergräbt genau dieses Prinzip.
**Fix:** CLAUDE.md + `docs/ai/*` an den realen Stand angleichen (Auth-Architektur, Endpunktliste, Testzahlen, CI-Status). Ggf. ADR für die Auth-/Patienten-Erweiterung nachziehen.

#### M-5 — Kein KI-/Haftungs-Disclaimer in Berichten & PDF
Generierte Befunde/PDFs enthalten keinen sichtbaren Hinweis „KI-generierter Entwurf — fachliche Prüfung erforderlich". Für ein Tool, das medizinische Dokumente erzeugt, ist das fachlich und rechtlich geboten.
**Fix:** Pflicht-Disclaimer in PDF-Footer und UI; Berichte als „Entwurf" kennzeichnen, bis vom Therapeuten freigegeben.

#### M-6 — Anamnese-Abschlusslogik inkonsistent
`missing_fields: []` bei gleichzeitig `is_anamnesis_complete: false` — die Engine fragt weiter, obwohl keine Pflichtfelder offen sind. Verwirrend für Nutzer (kein klares „fertig"-Signal).
**Fix:** Abschlusskriterium explizit definieren (Pflichtfeld-Set pro Berichtstyp) und konsistent an `is_anamnesis_complete` koppeln; UI-„Bericht erstellen"-Button aktivieren, sobald erfüllt.

### 🟢 NIEDRIG

- **L-1 — Stale E2E-Tests:** `landing.spec.ts` sucht „Wie es funktioniert" / „Alle Features"; reale Copy: „Vom Gespräch zum Bericht in 3 Schritten" etc. Tests beim Copy-Rewrite nicht mitgepflegt. → Tests an aktuelle Headings angleichen.
- **L-2 — Modell-Liste teils veraltet:** `_JSON_MODELS` enthält `llama-3.1-70b-versatile` (bei Groq dekommissioniert). Fallback fängt es ab, aber die Primärliste sollte aktuelle Modelle führen. → Liste pflegen.
- **L-3 — Material-Truncation:** Uploads werden bei 2000 Zeichen abgeschnitten ([report_generator.py:156](../../backend/services/report_generator.py#L156)) — bei längeren Vorbefunden Informationsverlust. → Chunking/Summarisierung statt hartem Cut.
- **L-4 — `/health` hinter SERVICE_TOKEN:** Bewusstes Design, aber untypisch — externe Uptime-Monitore/Load-Balancer können es ohne Token nicht prüfen. → ggf. einen ungeschützten Liveness-Ping (`/livez`) ergänzen.
- **L-5 — Feld-Naming-Inkonsistenz:** API mischt `production`/`produced`, `realname`/`first_name+last_name`; das untypisierte Phonologie-Schema verdeckt das. → Schemata konsolidieren + dokumentieren.
